> 마지막 업데이트: 2026-02-11

## 1. 문체 (Tone)

### 존댓말 사용 원칙
- 기본: 존댓말 (~습니다/~입니다) 사용
- 코드 설명: 존댓말 유지 (예: "설정에서 파일 시스템 캐싱을 활성화하세요")
- 인용/직접 화법: "~했습니다" 형태 유지 (예: "Horthy는 설명했습니다.")

### 문장 종결 패턴
- **서술문**: "~입니다", "~합니다", "~됩니다"
  - 예: "Turbopack이 개발 및 프로덕션 빌드 모두에서 안정화되었습니다."
- **권유문**: "~하세요", "~주세요"
  - 예: "설정에서 파일 시스템 캐싱을 활성화하세요."
  - 예: "피드백을 공유해주세요."
- **설명문**: "~합니다", "~됩니다"
  - 예: "이 API는 사용자 인터랙션이 필요한 기능에서 변경사항이 즉시 반영되도록 보장합니다."
- **목록 항목**: 명사형 또는 간결한 서술문
  - 예: "2-5배 빠른 프로덕션 빌드"
  - 예: "최대 10배 빠른 Fast Refresh"

### 자연스러운 톤
- **약간의 구어체 허용**: 기술 문서도 딱딱하지 않게 작성
  - 예: "따로 수정 같은 거 없이도 모든 환경에서 작동합니다" (자연스러움 향상)
- **인용된 사용자 질문**: 실제 사용자가 물어볼 법한 톤 사용
  - 예: "이 통합된 환경에서 뭐를 어떻게 하나요?" (기술적 표현보다 자연스러움)

## 2. 번역투 회피 패턴

### 반드시 피해야 할 표현

| 번역투 | 자연스러운 표현 | 예시 |
|--------|----------------|------|
| "~하는 것이 가능하다" | "~할 수 있다" | ❌ 실행하는 것이 가능하다 → ✅ 실행할 수 있다|
| "~에 의해 ~되다" | 능동태로 변환 | ❌ 팀에 의해 개발되었다 → ✅ 팀이 개발했다|
| "그것은/이것은" | 주어 구체화 또는 생략 | ❌ 이것은 중요한 기능이다 → ✅ 중요한 기능이다|
| "~경우" 과다 | "~하면", "~할 때" 등으로 변환 | ❌ 에러가 발생하는 경우 → ✅ 에러가 발생하면 |
| "~에 대해" 과다 | 조사로 대체 | ❌ 기능에 대해 설명 → ✅ 기능을 설명 |
| "~것이다" 과다 | "~합니다"로 변환 | ❌ 개선될 것이다 → ✅ 개선됩니다 |
| "제공하다" 과다 | 명확한 동작 표현 | ❌ 정보를 제공한다 → ✅ 정보를 담는다/전달한다/포함한다 |
| "갖추고 있습니다" | "제공합니다" | ❌ 기능을 갖추고 있습니다 → ✅ 기능을 제공합니다 |
| 부자연스러운 외래어 | 자연스러운 한글 표현 | ❌ 패키징되다 → ✅ 구성되다 |
| "이들은/이것은" 과다 | 구체적 표현으로 직접 대체 | ❌ 이들은 목표값입니다 → ✅ 아래는 대략적 지표입니다 |
| "~하고 있습니다" | "~중입니다" 간결화 | ❌ 개발하고 있습니다 → ✅ 개발 중입니다 |
| "~해야 하는지" | "~할지"로 간결화 | ❌ 사용해야 하는지 → ✅ 사용할지 |
| 추상적 기술 용어 | 구체적 의미로 풀어 설명 | ❌ 점진적 공개의 첫 번째 수준 → ✅ 컨텍스트에 가장 먼저 포함되는 정보 |
| "raw" 직역 | 맥락에 따라 생략 또는 명확화 | ❌ 원시 도구 접근 → ✅ 도구 접근 |
| "~면" vs "~하여" | 수단/조건 구분 | ❌ 스킬을 사용하면 Claude에게 → ✅ 스킬을 사용하여 Claude에게 |
| "~할 수 있습니다" 과다 | 다양한 표현으로 변형 | ❌ 5번 반복 → ✅ "~됩니다", "~가능합니다", "~하세요" 등으로 변형 |
| "필요에 따라" | "필요에 맞춰", "상황에 맞게" | ❌ 필요에 따라 진행할 수 있습니다 → ✅ 필요에 맞춰 다양한 수준으로 진행할 수 있습니다 |
| "반복 개선" | "계속 개선", "지속적으로 개선" | ❌ 반복 개선하세요 → ✅ 계속 개선하세요 |
| "~을 바탕으로 볼 때" | 간결한 연결어 | ❌ 초기 경험을 바탕으로 볼 때 → ✅ 초기 경험에서 |
| 명사형 나열 (추가 - 포함) | 동사형 문장화 | ❌ 추가 - 에러 핸들링 추가 → ✅ 에러 핸들링을 추가하세요 |
| "만났다면" (기술 문맥) | "발견했다면" | ❌ 엣지 케이스를 만났다면 → ✅ 엣지 케이스를 발견했다면 |
| "다시 입력하세요" (피드백 맥락) | "피드백하세요" | ❌ 다시 입력하세요 → ✅ 피드백하세요 |
| "찾아낸" | "발견한" | ❌ 찾아낸 이슈 → ✅ 발견한 이슈 |
| "~에 관해 작성할 때" | "~을 작성할 때" | ❌ 스킬에 관해 작성할 때 → ✅ 스킬을 작성할 때 |
| "~해야 한다고 믿습니다" | "~해야 한다고 생각합니다" | ❌ Claude가 도움이 된다고 믿습니다 → ✅ Claude가 도움이 된다고 생각합니다 |
| "고무적입니다" | "매우 긍정적입니다" | ❌ 고무적입니다 → ✅ 매우 긍정적입니다 |
| "결정됩니다" (피동형) | "좌우합니다" (능동형) | ❌ 스킬의 성공이 결정됩니다 → ✅ 스킬의 성공을 좌우합니다 |
| "필요로 할 때" | "거쳐야 할 때" 능동화 | ❌ 프로세스를 필요로 할 때 → ✅ 여러 단계를 거쳐야 할 때 |
| "~하는 것을 고려하세요" | 직접 권고 | ❌ 번들링하는 것을 고려하세요 → ✅ 번들링하는 방식을 추천합니다 |
| "참조하세요" | "확인하세요" | ❌ 섹션을 참조하세요 → ✅ 섹션을 확인하세요 |
| "기울어집니다" | "치우쳐 있습니다" | ❌ 한쪽으로 기울어집니다 → ✅ 한쪽으로 치우쳐 있습니다 |
| "보여줍니다" (과다) | 문맥에 맞는 동사 | ❌ 결과를 보여줍니다 → ✅ 결과를 내보냅니다/담습니다 |
| "접근하다" (과다) | "얻다", "접근하다" (상황에 맞게) | ❌ 마크다운에 접근할 수 있다 → ✅ 마크다운을 얻을 수 있다 |

### 외래어 대신 한국어 선호 패턴

|--------|------------|------|
| "뉘앙스" | "맥락", "의미", "어감" | ❌ 뉘앙스가 다름 → ✅ 맥락이 다름 |
| "스위트(suite)" | "모음", "세트" | ❌ evaluation suite → ✅ 평가 모음 |
| | | ❌ test suite → ✅ 테스트 모음 |

### 구조적 번역투 회피

| 번역투 구조 | 자연스러운 구조 | 예시 |
|-----------|--------------|------|
| "복수형 주어 + ~은" | 주어 전환 또는 단순화 | ❌ 가장 효과적인 스킬 제작자들은... → ✅ 가장 효과적인 방법은, ...하는 것입니다 |

### 문화적 로컬라이제이션

| 원문 표현 | 로컬라이제이션 | 이유 |
|-----------|--------------|------|
| Home Depot | 철물점 | 한국 독자에게 낯선 미국 매장 → 친숙한 비유로 교체 |

**원칙**: 원문의 비유/예시가 한국 독자에게 낯설 때, 동일한 맥락을 전달하는 한국식 비유로 교체

### 크로스 레퍼런스 확인 (시리즈 번역 시)

- 원본 PDF는 단일 문서이므로 내부 앵커 링크(`#section-name`)가 작동
- 블로그 시리즈로 분리하면 파트 간 참조가 **깨짐**
- **필수 확인**: 번역 시 내부 링크가 다른 파트를 가리키는지 확인 → 해당 파트의 블로그 URL로 수정
- 예: `#the-description-field` → `/posts/claude-skills-guide-part-2/#description-필드`

## 3. 용어 규칙

### 용어집 참조
@.claude/skills/translate-writer/data/glossary.md

### 원문 유지 용어
샘플에서 확인된 원문 유지 용어:
- **프레임워크/도구**: React, Next.js, TypeScript, Turbopack, Babel, ESLint, Tailwind CSS
- **기능명**: Fast Refresh, App Router, View Transitions, Cache Components
- **API/함수**: `revalidateTag()`, `updateTag()`, `refresh()`, `useEffectEvent()`, `<Activity/>`
- **설정 키**: `experimental.cacheComponents`, `images.localPatterns`
- **약어**: SWR (Stale-While-Revalidate), SDK, API, TTL, PPR
- **고유명사**: Y Combinator, YC, Claude Code, Claude Agent SDK, Vercel

### 한글화 용어
샘플에서 확인된 한글화 용어:
- **일반 기술 용어**:
  - bundler → 번들러
  - caching → 캐싱
  - routing → 라우팅
  - navigation → 네비게이션
  - prefetching → 프리페칭
  - memoization → 메모이제이션
  - revalidation → 재검증 (괄호 병기: revalidation)
  - middleware → 미들웨어
  - deduplication → 중복 제거
  - dependency → 의존성 (한국 개발자 커뮤니티 통용어)
  - suite → 모음 (evaluation suite → 평가 모음, test suite → 테스트 모음)
- **React/Next.js 용어**:
  - component → 컴포넌트
  - rendering → 렌더링
  - hydration → 하이드레이션
  - layout → 레이아웃
  - streaming → 스트리밍
- **일반 용어**:
  - production → 프로덕션
  - development → 개발
  - deprecated → 더 이상 사용되지 않는 / 지원 종료 예정
  - breaking changes → 주요 변경사항
  - migration → 마이그레이션

## 4. 구조 규칙

### 필수 요소
- **번역 안내 blockquote**: `> 이 문서는 [원문 제목]의 한글 번역입니다.`
  - 링크 포함 형식: `> 이 문서는 [원문 제목](URL)의 한글 번역입니다.`
- **목차**: `## 목차` (Astro가 자동 생성)
- **TL;DR**: `<details>` 태그로 접을 수 있게
- **원문 정보**: 작성자, 작성일 (구분선 후 배치)
- **참고 자료**: `## 참고 자료` (문서 마지막에 배치)

### 섹션 순서 (샘플 기준)
1. Frontmatter (YAML)
2. 번역 안내 blockquote
3. 목차 (`## 목차`)
4. 핵심 요약 (`## 핵심 요약` + TL;DR)
5. 구분선 (`---`)
6. 원문 정보 (작성일, 작성자)
7. 본문 서론 (원문 내용 시작)
8. 본문 섹션들
9. 참고 자료 (`## 참고 자료`)

### TL;DR 형식
```markdown
## 핵심 요약

<details>
<summary><strong>📌 TL;DR (클릭하여 펼치기)</strong></summary>

### 주요 내용/개선사항
- [핵심 포인트 1]
- [핵심 포인트 2]
- [핵심 포인트 3]

### 주의사항/핵심 메시지 (선택)
- [주의할 점 또는 핵심 메시지]

</details>
```

### 원문 정보 형식
```markdown
---

**원문 작성일**: YYYY년 MM월 DD일 (요일)

**작성자**: [원문 작성자 이름]
```

### 참고 자료 형식
```markdown
## 참고 자료

- [제목](URL)
- [원문 제목 - 원문](원문 URL)
```

## 5. Frontmatter 규칙

```yaml
---
title: "[번역] 원문 제목"  # [번역] 말머리 필수
description: "[번역] 또는 원문 제목 + 핵심 내용 요약 120-160자"
pubDatetime: YYYY-MM-DDTHH:MM:SSZ  # ISO 8601 UTC (date -u 명령으로 확인)
modDatetime: YYYY-MM-DDTHH:MM:SSZ  # 수정일 (선택)
featured: false  # 또는 true
draft: false  # 또는 true (작성 중일 때)
tags: ["translation", "관련-태그", ...]
---
```

### pubDatetime 설정 주의사항

> ⚠️ **중요**: `pubDatetime`이 미래 날짜면 Astro 빌드에서 글이 제외됩니다!

- `pubDatetime`은 **실제 현재 UTC 시간**으로 설정
- 시스템 프롬프트의 날짜가 아닌 **`date -u +"%Y-%m-%dT%H:%M:%SZ"`** 명령 결과 사용
- 한국 시간(KST)은 UTC+9이므로, KST 자정은 UTC 전날 15:00임을 주의

### title 패턴
- **필수 형식**: `"[번역] 원문 제목"`
- 예시: `"[번역] Next.js 16 Beta"`, `"[번역] 세 YC 스타트업이 Claude Code로 회사를 구축한 방법"`

- 제목 반복 + 핵심 키워드 나열

## 8. 문장 구조 및 연결

### 짧은 문장 병합
- **패턴**: 두 개의 짧은 문장을 "~며", "~고", "~면서" 등으로 자연스럽게 병합
- ❌ "간단한 폴더 형태로 구성됩니다. 스킬은 Claude를"
- ✅ "간단한 폴더 형태로 구성되며 Claude를"
- **교훈**: 불필요한 문장 분리 제거 → 가독성과 자연스러움 향상

### 조건과 수단 구분
- **"~하면"**: 조건을 나타낼 때 (If 뉘앙스)
  - "에러가 발생하면 다시 시도하세요"
- **"~하여"**: 수단/방법을 나타낼 때 (By means of 뉘앙스)
  - "스킬을 사용하여 Claude에게 지시사항을 전달합니다"
- ❌ "스킬을 사용하면 Claude에게" (수단이 아닌데 ~하면)
- ✅ "스킬을 사용하여 Claude에게" (수단을 명확하게)

### 관계절과 동격 표현
- **기술 문맥에서는 동격을 명확히**: 관계절 제거로 간결화
- ❌ "Next.js 라우트 핸들러로 요청을 전달하는 Contentful 콘텐츠를 마크다운으로 변환합니다"
- ✅ "Next.js 라우트 핸들러가 Contentful 콘텐츠를 마크다운으로 변환합니다" (주어 명확화)
- **원칙**: 관계절이 복잡하면 주어를 명시하여 의미를 명확하게 함

## 10. 역주(Translator's Note) 규칙

### 역주 추가 시점
- **개념 설명 필요**: 한국 독자에게 낯선 기술 용어나 표준을 소개할 때
  - 예: RFC 표준 링크, HTTP 개념 설명
- **문화적 배경**: 원문의 비유나 사례가 한국 독자에게 낯설 때
  - 예: Home Depot → 철물점 (필요시 역주로 추가 설명)
- **번역 선택 설명**: 특정 용어를 의도적으로 번역하거나 원문 유지한 이유 설명

### 역주 형식
```markdown
> **역주**: [설명 내용]

> **역주**: 콘텐츠 협상(Content Negotiation)은 [RFC 7231](URL)에 정의된 HTTP 표준 메커니즘입니다. 클라이언트가 `Accept` 헤더를 통해 "나는 이런 형식을 선호합니다"라고 알리면, 서버가 같은 URL에서 요청에 맞는 최적의 형식을 골라 응답합니다.
```

### 역주 배치
- 본문 서론 직후, 해당 개념이 처음 등장할 때 추가
- 너무 많은 역주는 가독성을 해치므로 필수적인 경우만 추가 (1-2개 권장)
