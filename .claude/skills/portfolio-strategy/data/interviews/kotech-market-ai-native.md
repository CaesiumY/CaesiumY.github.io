## 답변 정리

### 앵커링

- **기간·역할**: 한국 기술 마켓 프로젝트(2024.05~2026.05, 2년) 중 본인이 kotech-plugin을 처음부터 설계·운영. 외주 회사의 개발 프로세스에 최적화된 자체 도구로 자리 잡음.
- **정량 지표 근거**: "3주 → 3일"은 본인 경험 기반 비교. 이전에 동일 분량의 모노레포 앱을 피그마대로 만든 작업이 3주 소요됐고, AI-Native 방식 도입 후 3일 수준으로 축소. 사내 측정 도구가 아닌 본인 작업 시간 기준 추정.
- **기술 도입 강도**: kotech-plugin 자체를 본인이 직접 설계·구현. TailwindCSS 기반 디자인 시스템, 피그마 MCP·devmode 혼합, SSOT JSON, 스토리북, GitHub 이슈 자동화 모두 본인 주도.

### 정량 (Result 슬롯)

- **신규 브랜드 출시 공수**: 3주 → 3일 (본인 경험 기반)
- **디자인 시스템 만족 결과 시도 횟수**: 테스트 샘플 페이지 단독 방식 20회 시도 → 스토리북 도입 후 약 2회 만에 만족 결과 (약 10배 효율 개선)
- **적용된 레포 수**: 3개 (유저 모노레포 8+ 앱 / 관리자 / 대시보드)
- **이슈 보고/해결 사이클**: 시간보다 "번거로움" 정성 개선 — 매번 에이전트에 말로 설명하던 워크플로우가 GitHub 이슈 기반으로 전환되어 문제 파악·작업 트래킹이 용이해짐

### 의사결정·트레이드오프

- **대안 검토**: 외주 회사 프로세스에 최적화된 미래 재사용 자산이 필요했고, 일회성 셸 스크립트나 단순 템플릿 리포지터리는 반복 작업 장기 효율화에 부족하다고 판단.
- **채택 이유**: "지금뿐 아니라 앞으로도 반복될 작업"이라는 미래 재사용성 + 외주 워크플로우 최적화. 플러그인 형태가 가장 적합.
- **포기한 것 (Trade-off)**: 별도 레포로 분리하면서 독립성은 향상되었으나, 테스트 플라이트에서 발견된 이슈를 곧바로 적용하기 어려운 문제 발생 (다른 레포에서 수정해야 함).
- **보완 방법**: "이슈 보고 스킬 + 이슈 해결 스킬" 양방향 자동화 — 테스트 플라이트 결과 보고에서 발견된 이슈를 플러그인 레포에 GitHub 이슈로 자동 발행, 플러그인 레포에서 이슈 해결 스킬을 호출해 수정 진행.

### 실패·재발견

- **첫 시도 실패**: 테스트용 샘플 페이지를 통한 디자인 시스템 검증 방식. 20번 이상 테스트 플라이트를 시도했으나 위젯 누락, 피그마 MCP 미호출 등 비결정성이 심해 멱등성 보장이 어려웠음.
- **전환점 1 (자가 점검 루프)**: SSOT 방식으로 진행 과정을 JSON에 기록 → 작업 종료 후 누락 내용을 스스로 파악하고 재진행하는 자가 점검 루프 도입.
- **전환점 2 (스토리북 추상화)**: 디자인 시스템 자체의 비결정성이 여전히 남자 추상화 레이어로 스토리북을 도입. 약 2회 만에 만족할 만한 결과 도출.
- **숨은 제약 (피그마 MCP 한도)**: 피그마 MCP는 한도가 있어 devmode와 혼합 사용해야 함. 이로 인해 파운데이션 → 빌드 2중 구조 정립.
  - 파운데이션: 모노레포 앱 생성 + 흩어진 피그마 링크 따라가서 노드 ID 등 SSOT JSON으로 저장(멱등성 보장) → 노드 ID로 피그마 MCP 호출해 디자인 시스템 생성 → 스토리북 검증
  - 빌드: 각 라우트의 PRD 생성 → 서브 에이전트로 독립 환경에서 병렬 페이지 조합
