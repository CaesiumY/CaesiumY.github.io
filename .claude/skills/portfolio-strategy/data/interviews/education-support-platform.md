## 답변 정리

### 앵커링

- **기간·역할**: 8개월(2025.05~2026.01)은 유지보수 포함 계약 기간. 실제 개발 단계는 **예상 2개월 → 실제 1개월**로 단축. 본인이 납품 시점 기준으로 직접 측정.
- **정량 지표 근거**: AI 기반 작업 적용 후 예상 공수 2개월이 1개월로 줄어듦. 사내 측정 도구가 아닌 본인 측정 기준 (작업 마치고 납품한 시점).

### 정량 (Result 슬롯)

- **개발 공수**: 예상 2개월 → 실제 1개월 (50% 단축, 본인 측정)
- **계약 기간**: 8개월 (개발 후 유지보수 포함)

### 의사결정·트레이드오프

- **MSW 채택 이유**: 가장 많이 쓰이는 라이브러리 + Orval과 연동 가능한 점.
- **Orval 채택 이유**: 다른 라이브러리(예: openapi-typescript-codegen, swagger-codegen 등)와 비교했고, **Orval은 TanStack Query 코드까지 모두 자동 생성**해주는 점이 React 기반 프로젝트에 가장 적합하다고 판단.
- **포기한 것 (Trade-off)**: Orval 자동 생성 코드의 **응집성** — API 코드가 각 features/domain 폴더에 위치할 수 없고 자동 생성 폴더 내부에 집중됨.
- **트레이드오프 정당화**: "응집성도 결국 개발자가 개발을 용이하게 하기 위한 수단이다. 자동 생성되는 코드는 개발자가 신경 쓸 필요가 없는 코드이기에 응집성을 포기해도 된다. 목표와 수단을 구분해야 한다." (목표·수단 구분 사고)

### 협업·갈등 조율 (Q19)

- **사건**: 백엔드 팀이 사전 협의 없이 ID를 숫자형으로 수정해 달라고 요청.
- **상황**: 본인이 이미 MSW로 선제 스키마를 구현해 놓은 상태였음.
- **역제안**: "ID를 숫자형으로 해야 하는 특별한 이유라도 있느냐"고 질문하며, **JS의 Max Integer 한계로 장기 이슈 가능성** 근거 제시.
- **의의**: MSW 선제 스키마 보유가 단순 일정 단축이 아니라 **백엔드 결정에 데이터 리스크 근거를 즉시 제시할 수 있는 협상 자산**으로 기능. 토스 인재상 "데이터 흐름·상태 변화 이해 + 비즈니스 맥락" 매칭.
