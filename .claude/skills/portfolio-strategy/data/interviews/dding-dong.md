## 답변 정리

### 앵커링

- **기간·역할**: 사이드 프로젝트 (2026.02). 본인 단독 설계·구현·배포.

### 문제 (P 보강)

- 다중 Claude Code 세션 병행 시 **CLI에 알림이 없어** 여러 터미널을 왔다갔다해야 함.
- 병행 작업이 끝나거나 질문이 있을 때 세션 스위칭이 어려움.
- 다음 작업을 곧바로 지시하려면 **항상 컴퓨터 앞에 앉아 있어야** 했음.

### 의사결정·트레이드오프

- **검토한 대안**: 슬랙·디스코드 등 일반 메신저 기반 알림 — **회피** (이미 시장에 많은 솔루션).
- **채택 이유**:
  1. Claude Code의 훅 시스템을 활용해 시장에 없는 차별화된 알림 시스템 구축
  2. **토이 프로젝트로 재미있게 진행**한다는 의도 (진정성 있는 사이드 프로젝트)
  3. **Qwen Voice 모델로 목소리 클론**해 본인만의 알림 — 차별화 요소
- **사용자 경험 의사결정**: 진입 장벽 최소화를 위해 **마켓플레이스 + 플러그인 + 스킬 + 랜딩 페이지 통합 배포** 선택.

### 정량 (Result 슬롯) — 사이드 프로젝트 정성 표현

- **정량 지표 부재**: 사이드 프로젝트라 비즈니스 정량 지표는 없음. 솔직하게 정성 표현으로 정리.
- **본인 생산성**: 입력 요청을 놓치고 있는 시간이 줄어듦.
- **학습 가치**:
  - Qwen Voice 활용 경험
  - OS별 CLI 알림 솔루션 차이 인지 (macOS/Linux/WSL 각기 다름)
- **운영 가치**: 마켓플레이스 배포 + 랜딩 페이지를 통한 사용자 진입 경험 설계

### 까다로웠던 의사결정 (Q3 답변)

- **크로스 플랫폼 알림**이 가장 까다로웠음 — OS별 CLI 알림 솔루션이 달랐기 때문.
- **사용자 세팅 편의성** — 어떻게 해야 사용자가 쉽게 세팅해서 사용할 수 있을지 고민.
- 결과적으로 **"마켓플레이스 + 플러그인 + 스킬"** 조합 + **랜딩 페이지** 입문 동선 구성.
