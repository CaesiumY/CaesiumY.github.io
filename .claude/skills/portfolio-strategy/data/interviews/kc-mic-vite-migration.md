## 답변 정리

### 앵커링

- **기간·역할**: KC-MIC 프론트엔드 엔지니어 (2022.03~2023.09, 1년 6개월) 중 본인 주도 마이그레이션
- **정량 지표 근거**: "70%"는 **빌드 명령어 실행 후 출력되는 숫자로 실측**. 사내 측정 도구가 아닌 빌드 출력 기준 비교.

### 정량 (Result 슬롯)

- **빌드 시간 단축**: 70% (실측, 빌드 명령어 출력 기준)

### 문제 (P 보강)

- 당시 CRA 환경이 **eject로 웹팩 설정을 노출 + craco로 덮어씌우는 비표준 구조**였음.
- **설정 추적 어려움** + **개발 서버/빌드 속도가 현저히 느려 DX에 직접 영향**.
- 즉, 단순 "레거시"가 아니라 "비표준 빌드 환경의 누적된 기술 부채"가 문제의 본질.

### 의사결정·트레이드오프

- **검토한 대안**:
  1. Vite로 마이그레이션
  2. 웹팩 설정을 되돌려 **순정 CRA로 회귀**
- **채택 이유**: 속도 이슈 해결이 우선 과제였고, 당시 가장 빠르다고 평가받고 있었으며 생태계에서 떠오르고 있던 Vite 선택.
- **마이그레이션 방식**: **한 번에 진행** (점진 X). Vite 마이그레이션 자체가 생각보다 쉬웠음. → 이 점은 토스 우대 키워드 "점진적 이관"과 직접 매칭은 약하지만, 비표준 craco 환경 정리라는 측면에서 "기술 부채 관리" 인재상과 매칭.

### 협업 (함수형 전환 합의)

- **상황**: 클래스 컴포넌트→함수형 전환에 대해 팀 내 의견 차이
- **설득 근거**:
  1. 기본 제공 Hook의 유용성
  2. 앞으로의 React 발전 방향 (함수형이 표준이 될 흐름)
- **결과**: 팀 합의 도출
