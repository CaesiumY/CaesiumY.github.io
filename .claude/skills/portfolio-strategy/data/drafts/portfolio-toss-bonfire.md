### §1. 한국 기술 마켓 · 프론트엔드 엔지니어 · AI-Native 개발 환경 구축 (2년)

**문제**

- 다중 브랜드 확장 시 동일 패턴 반복 작업이 신규 브랜드 한 개당 약 3주씩 소요되어 외주 워크플로우의 핵심 병목이 되었다.
- 테스트 샘플 페이지 단독 검증 방식은 위젯 누락·MCP 미호출 등 비결정성이 심해, 20번 이상 시도해도 멱등성 보장이 어려웠다.

**해결**

- 파운데이션→빌드 2중 구조 파이프라인을 설계했다. 파운데이션은 흩어진 피그마 노드 ID를 SSOT JSON으로 저장해 멱등성을 보장하고, 빌드 단계는 라우트별 PRD를 생성한 뒤 서브 에이전트가 독립 환경에서 병렬로 페이지를 조합한다.
- 디자인 시스템 검증 레이어로 스토리북을 도입했다. 단순 검증 도구가 아니라 추상화 레이어로 작동시켜 AI 출력 비결정성을 통제하는 것이 목적이었다.
- 별도 레포 분리로 즉시 적용이 어려운 트레이드오프는 GitHub 이슈 보고/해결 양방향 자동화 스킬로 보완했다.

**결과**

- 신규 브랜드 출시 공수 **3주 → 3일**, 디자인 시스템 만족 결과 시도 횟수 **20회 → 약 2회**.
- 3개 레포(유저 모노레포 8+ 앱·관리자·대시보드) 운영의 핵심 인프라로 자리 잡았다.

### §2. 교육 지원 플랫폼 · 프론트엔드 리드 · MSW + Orval 파이프라인 (1개월 개발 + 7개월 유지보수)

**문제**

- 클라이언트의 잦은 기획 변경 환경에서 백엔드 API 설계가 매번 프론트엔드 진행을 가로막았다. API 미완성 상태로는 화면 검증·UI 분기 테스트도 불가능했다.

**해결**

- MSW로 API 스키마를 프론트엔드에서 선제 정의해 백엔드에 역제안하는 워크플로우를 도입했다. 이는 일정 단축뿐 아니라 백엔드 결정의 근거 검증 자산으로 작동했다 — 백엔드가 사전 협의 없이 ID를 숫자형으로 변경 요청했을 때 JS Max Integer 한계를 근거로 즉시 역제안할 수 있었다.
- Orval로 OpenAPI 변경 시 클라이언트 SDK가 자동 재생성되도록 했다. TanStack Query 통합이 기본 제공되어 데이터 페칭 보일러플레이트까지 흡수하기 때문이다.
- 자동 생성 코드의 응집성 트레이드오프는 "응집성도 개발 용이성을 위한 수단이며 자동 생성 코드는 신경 쓸 필요가 없다"는 목표·수단 구분 원칙으로 정당화했다.

**결과**

- 개발 공수 **2개월 → 1개월** (50% 단축, 본인 측정), 유지보수 포함 **8개월 계약 완수**.

### §3. KC-MIC · 프론트엔드 엔지니어 · 빌드 환경 현대화 (1년 6개월)

**문제**

- 당시 CRA가 eject로 웹팩 설정을 노출 + craco로 덮어씌운 비표준 환경이었다. 설정 추적이 어려웠고, 무엇보다 개발 서버·빌드 속도가 현저히 느려 DX에 직접 영향을 주고 있었다.

**해결**

- 검토한 두 대안은 (1) Vite 진보, (2) 웹팩 설정을 되돌려 순정 CRA로 회귀였다. 속도 이슈 해결을 우선 목표로 두고, 가장 빠르다고 평가받고 생태계에서 떠오르고 있던 Vite를 채택했다.
- Vite 마이그레이션 자체가 생각보다 쉬워 한 번에 진행했다. 누적된 craco 비표준 환경 정리가 동시에 이루어지면서 기술 부채를 함께 해소했다.
- 클래스 컴포넌트 → 함수형 전환은 Hook 유용성과 React 발전 방향을 근거로 팀 합의를 도출했다.

**결과**

- 빌드 시간 **70% 단축** (빌드 명령어 출력 기준 실측). 비표준 빌드 환경 정리로 DX 회복.

### §4. dding-dong · 사이드 프로젝트 · 오픈소스 Claude Code 알림 플러그인 (2026.02)

**문제**

- 다중 Claude Code 세션을 병행할 때 CLI에 알림이 없어 여러 터미널을 왔다갔다해야 했다.
- 다음 작업을 곧바로 지시하려면 항상 컴퓨터 앞에 상주해야 했다.

**해결**

- 이미 시장에 많은 슬랙·디스코드 메신저 방식 대신 Claude Code 훅 시스템에 직접 통합한 차별화된 알림을 설계했다. 토이 프로젝트의 재미를 살려 Qwen Voice 모델로 목소리를 클론한 TTS 커스텀 팩을 더했다.
- 크로스 플랫폼 CLI 알림 솔루션이 OS별로 달라 추상화 레이어를 별도 설계했다.
- 사용자 진입 장벽 최소화를 위해 마켓플레이스·플러그인·스킬·랜딩 페이지를 통합 배포 묶음으로 구성했다.

**결과**

- **3개 OS 지원**(macOS/Linux/WSL)과 **4개 진입점**(마켓플레이스·플러그인·스킬·랜딩 페이지) 통합 배포로 사용자 진입 장벽 최소화.
- 본인 워크플로우 측면에서는 다중 세션 입력 요청 누락이 줄어 컴퓨터 앞 상주 부담이 해소되었다.

### §5. AI 강의 슬라이드 자동 생성 파이프라인 · 사이드 프로젝트 (2026)

**문제**

- 14주차 정규 교과목 슬라이드 제작은 회차당 디자인 직접 + 내용 전달 깊이 고민이 누적되어 시간 소모가 컸다. 외주 견적은 약 2주였다.

**해결**

- "AI가 아니더라도 인간이 진행하는 단계를 그대로 옮겨온다"는 원칙으로 다단계 파이프라인을 설계했다. 매주차 일관성 보장을 위해 디자인 시스템을 우선 구축한 뒤 슬라이드 생성 단계로 진입했다.
- 각 슬라이드의 정보와 화면 구성(그래프·숫자 비교 등)을 PRD 형식으로 문서화한 뒤 생성 단계로 넘겼다.
- 커스텀 폰트의 텍스트 박스 가변 크기로 HTML 미리보기와 PPTX 완성본 사이 괴리감이 발생했다. 스크린샷 비주얼 검증 단계를 추가해 출력 일관성을 통제했다.
- 이미지가 아닌 PPTX 형식을 유지한 이유는 강의 촬영 현장의 즉석 수정 리스크를 회피하기 위함이었다.

**결과**

- 견적 **2주 → 실제 3일** (14주차 전체 분량). 이후 외부 업체 특강에도 동일한 파이프라인을 활용했다.
