# 포트폴리오 마크다운 템플릿

> 이 파일은 Phase 3(작성)에서 `portfolio-strategist` 에이전트가 따르는 출력 형식을 정의한다. 모든 출력은 한국어 마크다운이며, 사용자가 직접 `src/pages/portfolio.md`에 옮겨 다듬는 것을 전제로 한다.

## 1. 도입부 템플릿 (`motivation`)

포트폴리오 최상단에 한 번만 배치한다. 본문 섹션이 추가될 때마다 다시 생성하지 않는다.

```markdown
## 즉시 발휘 가능한 역량

[지원자 이름]은 [핵심 기술 도메인] 영역에서 [경력 연수] 동안 [핵심 성과의 한 문장 요약]을 만들어왔다. 팀에 합류하면 다음 역량을 곧바로 발휘할 수 있다.

- **[역량 1]**: [한 줄 근거 — 어떤 프로젝트에서 어떻게 입증되었는가]
- **[역량 2]**: [한 줄 근거]
- **[역량 3]**: [한 줄 근거]
- (선택) **[역량 4]**, **[역량 5]**

각 역량은 본문 섹션 [N]에서 상세히 풀어 설명한다.
```

### 길이 가드 (도입부)

- 첫 문단: 2~3 문장 (50~80 단어)
- bullet: 3~5개. 각 bullet은 한 줄 (10~25 단어). 두 줄 이상 늘어지면 본문 섹션으로 옮긴다.

### 작성 원칙 (도입부)

- "즉시 발휘 가능"의 의미는 신입 교육 기간 없이 바로 시작 가능한 역량이다. 학습 의향이나 잠재력이 아닌 검증된 실적을 적는다.
- 각 역량 bullet은 본문의 어느 섹션과 매핑되는지 1:1 명시. 채용 담당자가 도입부를 읽고 관심 가는 역량을 본문에서 즉시 찾을 수 있어야 한다.
- 추상적 형용사("열정적인", "능동적인")는 사용하지 않는다. 직무 역량의 동사·기술명·정량 지표로만 구성한다.

## 2. 본문 섹션 템플릿 (`generate` / `expand`)

이력서 한 줄당 한 섹션을 생성한다. 다음 구조를 엄격히 따른다.

```markdown
### [프로젝트명] · [역할 / 기간]

**문제**

- [원인 — 구체적인 상황과 제약. "어떤 시스템에서 어떤 일이 벌어지고 있었는가"]
- [연쇄 파급효과 — 그 원인이 어떤 비즈니스/기술 문제로 번졌는가]

**해결**

- [기술 적용 — 무엇을 사용해서 어떻게 해결했는가]
- (선택) [의사결정 근거 — 다른 선택지 대비 이걸 고른 이유]

**결과**

- [A를 적용하여 B라는 결과 + **N% 개선** 또는 정량 수치]
- (선택) [부수 효과 — 결과에서 파생된 추가 임팩트]
```

### 길이 가드 (본문 섹션)

- **문제**: 2~3 문장 또는 2 bullet
- **해결**: 3~5 문장 또는 2~3 bullet
- **결과**: 1~2 문장 + 정량 지표 1개 이상 필수
- 섹션 전체: 80~150 단어 권장. 200 단어 초과 시 평가 단계에서 가독성 감점.

### 작성 원칙 (본문 섹션)

- **결과 부분에 정량 지표가 빠지면 평가 자동 탈락**. 인터뷰 노트에서 사용자가 답한 수치(50% 단축, 70% 절감 등)를 반드시 결과에 인용한다. 인터뷰에서 수치를 얻지 못했다면 작성하지 말고 사용자에게 추가 질문한다.
- 문제 섹션은 "당시의 배경"이 아니라 "왜 이 작업이 필요했는가"의 답이다. 일반적인 사실 나열 대신 구체적 마찰을 묘사한다. 예: ❌ "프로젝트가 성장 중이었다" → ✅ "주간 빌드 실패율 30% — 매일 두 시간씩 디버깅에 소모"
- 해결 섹션은 도구 이름만 나열하지 않는다. "왜 그 도구를 그 방식으로 썼는가"가 의사결정 깊이를 드러낸다. 인터뷰의 트레이드오프 답변을 활용한다.

## 3. 풀 예시 (사용자 프로젝트 기반)

다음은 사용자의 `교육 지원 플랫폼` 프로젝트로 작성한 한 섹션 예시다. 인터뷰 노트에 기반한 가상의 답변을 사용했다.

```markdown
### 교육 지원 플랫폼 · 프론트엔드 리드 · 8개월 (2025.05~2026.01)

**문제**

- 클라이언트가 기획 변경을 잦게 요구하는 환경에서 백엔드 API 설계가 매번 프론트엔드 진행을 30% 가량 지연시켰다.
- API 미완성 상태로는 프론트엔드 화면 검증·UI 분기 테스트가 불가능해 QA 일정도 함께 밀렸다.

**해결**

- MSW(Mock Service Worker)로 API 스키마를 프론트엔드에서 선제적으로 정의하고 백엔드에 역제안하는 워크플로우를 도입했다. Cursor와 Claude를 파이프라인 핵심으로 활용해 스키마 변환·타입 생성을 자동화했다.
- Orval 도입으로 OpenAPI 변경이 발생하면 클라이언트 SDK가 자동 재생성되도록 했다. 단순 자동화 도구 대신 Orval을 고른 이유는 `react-query` 통합이 기본 제공되어 데이터 페칭 보일러플레이트까지 흡수했기 때문이다.

**결과**

- 예상 공수 16개월 대비 실제 8개월로 완료, **개발 기간 50% 단축**.
- 부수 효과로 프론트엔드 팀이 API 명세 주도권을 가지게 되어 클라이언트 기획 변경 시 협상 근거가 강화되었다.
```

### 예시 분석

위 예시를 평가 루브릭으로 점검하면:

- **구조 PASS**: 문제(원인+파급효과), 해결(기술+근거), 결과(50% 정량) 모두 존재
- **적합도**: 프론트엔드 직무 핵심 키워드(MSW, Orval, react-query, OpenAPI) + 협업 의사결정 모두 포함
- **가독성**: 142 단어, 권장 범위 내. 문장이 짧고 능동태

## 4. 출력 시 주의사항

- 각 섹션은 마크다운 문법 그대로 출력한다. 추가 메타데이터(점수, 평가 결과 등)는 출력에 포함하지 않는다 — 그것은 평가 단계 출력에서 별도 처리.
- 섹션 제목(`###`)은 이력서의 해당 한 줄과 매핑되어야 한다. 제목 옆에 역할·기간 메타를 작은 글씨가 아닌 평문으로 추가해 1:1 매핑을 시각적으로 강조한다 (예: `### 한국 기술 마켓 · 프론트엔드 엔지니어 · 2년`).
- 코드 블록·이미지·링크 사용은 자유롭게. 단, 링크가 외부 URL인 경우 한 번만 인용하고 본문 흐름을 끊지 않는다.
- 한국어 어투: 공식 보고서체 또는 정중한 평서체. 구어체("~했어요", "~인 것 같아요")는 사용하지 않는다.
