# Phase 4 평가 루브릭

> 이 파일은 Phase 4(평가)에서 `portfolio-reviewer` 에이전트가 따르는 채점 기준이다. IT 채용 담당자 1차 필터 시점이 기본 평가자다. `--reviewer=tech` 또는 `--reviewer=both` 옵션 시 기술 면접관 시점이 추가된다.

## 평가 모델 개요

채용 평가는 본질적으로 다차원이라 단일 점수보다 **카테고리별 임계 + 종합 점수** 모델을 사용한다. 한 카테고리가 심각하게 부족하면 다른 점수가 높아도 의미가 없기 때문이다 (예: 정량 지표가 빠진 포트폴리오는 가독성 100점이어도 결과 항목이 비어버린다).

| 카테고리 | 가중치 | 임계 | 검증 항목 |
|---|---|---|---|
| **구조 (Structure)** | 30 | **Binary 통과 필수** | 도입부 존재 (도입부 첫 작성 시), 문제·해결·결과 3구조 모두 존재, 결과에 정량 지표 ≥ 1개 |
| **적합도 (Relevance)** | 35 | **70점** | JD 핵심 키워드 매칭률, 직무 부합도, 도메인 맥락 적합성 |
| **가독성 (Readability)** | 35 | **75점** | 길이 적정성, 문장 흐름, 1차 스캔 가독성, 한국어 자연스러움 |

**PASS 조건**: 모든 카테고리 임계 동시 통과 **AND** 종합 가중평균 ≥ **80점**

## 카테고리 1: 구조 (Structure)

Binary 통과(0점 또는 100점). 다음 4개 항목 중 하나라도 빠지면 0점.

### 통과 조건

1. (도입부 첫 작성일 때만) **즉시 발휘 가능 역량 도입부**가 존재한다 — 한 단락 + 3~5 bullet.
2. **문제** 섹션이 존재하고, **원인**과 **연쇄 파급효과**가 모두 명시되어 있다.
3. **해결** 섹션이 존재하고, 사용된 기술이 명시되어 있다.
4. **결과** 섹션이 존재하고, **정량 지표(N%, N배, N분 등)가 1개 이상 포함**되어 있다.

### 통과 예시

```markdown
**문제**
- 클라이언트 기획 변경이 잦은 환경에서 백엔드 API 설계 지연이 프론트엔드 진행을 30% 정도 막았다.
- API 미완성 상태로는 화면 검증이 불가능해 QA 일정도 같이 밀렸다.

**해결**
- MSW로 API 스키마를 프론트에서 선제 정의하고 백엔드에 역제안. Orval로 OpenAPI 변경 시 자동 SDK 재생성.

**결과**
- 예상 공수 16개월 대비 8개월 완료, **개발 기간 50% 단축**.
```

→ 통과: 4개 항목 모두 존재.

### 미통과 예시

```markdown
**문제**
- API 설계가 오래 걸렸다.

**해결**
- MSW와 Orval을 도입했다.

**결과**
- 개발이 빨라졌다.
```

→ 미통과: 결과에 정량 지표 없음 ("빨라졌다"는 정량이 아님). 문제의 연쇄 파급효과 누락. 해결의 의사결정 근거 누락.

### 피드백 작성 가이드

- 어느 항목이 빠졌는지 명시: "결과에 정량 지표가 없습니다."
- 인터뷰 노트에서 활용 가능한 데이터 인용: "인터뷰 노트의 'Q6 답변: 빌드 시간 70% 단축'을 결과에 명시하세요."
- 추측 지시 금지: 인터뷰 노트에 데이터가 없으면 "사용자에게 추가 인터뷰를 요청해야 합니다"라고 보고한다.

## 카테고리 2: 적합도 (Relevance)

0~100점. 임계 70점.

### 채점 기준 (JD 제공 시)

JD가 제공된 경우 다음 4개 하위 항목으로 채점.

| 하위 항목 | 배점 | 평가 |
|---|---|---|
| 핵심 키워드 매칭 | 35 | JD에서 추출한 핵심 키워드(상위 5~7개) 중 자연스럽게 포함된 비율 × 35 |
| 직무 부합도 | 30 | JD 핵심 책임과 작성된 케이스의 유사도 (정성 평가, 0~30) |
| 도메인 맥락 | 20 | 회사·업계 도메인 용어와의 일치도 (예: 스타트업 vs 대기업, B2C vs B2B) |
| 시니어리티 매칭 | 15 | JD가 요구하는 경력 수준과 작성된 케이스가 보여주는 자율성·임팩트 수준의 일치 |

### 채점 기준 (JD 미제공 시)

일반 시장 표준으로 채점.

| 하위 항목 | 배점 | 평가 |
|---|---|---|
| 직무 핵심 역량 명시 | 40 | 프론트엔드라면 React/TS/성능, 백엔드라면 API/DB/스케일링 등 직무 핵심이 명시되었는가 |
| 한국 시장 표준 부합 | 30 | 일반적인 한국 IT 채용 시장에서 기대되는 케이스 스타일 (구체적 기술 + 비즈니스 임팩트) |
| 차별화 포인트 | 30 | 같은 경력자라면 누구나 적을 법한 내용을 넘어 본인 고유의 경험·관점이 드러나는가 |

### 통과 예시

JD에 "Turborepo 모노레포 경험 우대, FSD 아키텍처 경험 우대"가 있을 때:

```markdown
**해결**
- Turborepo 기반 모노레포로 다중 브랜드 앱을 단일 저장소로 통합. Core 패키지 중앙화로 신규 브랜드 출시 속도 향상.
- FSD 아키텍처 도입 후 생산성 저하를 발견 → 아키텍처는 목표가 아닌 수단으로 재정의, 커스텀 레이어 구조로 재설계.
```

→ Turborepo · FSD 모두 자연스럽게 등장 + 단순 도입이 아닌 의사결정 근거까지 포함. 매칭률 + 부합도 모두 높음.

### 미통과 예시

같은 JD에 다음과 같이 작성된 경우:

```markdown
**해결**
- 프론트엔드 개발을 진행했다. 다양한 기술을 활용했다.
```

→ JD 키워드 0개 매칭, 직무 부합도 측정 불가. 평가 25점 수준으로 미통과.

### 피드백 작성 가이드

- 누락된 키워드 명시: "JD의 'Turborepo' 키워드가 누락. 한국 기술 마켓 케이스에서 자연스럽게 언급."
- 키워드 스터핑 경고: 인위적으로 키워드를 박아넣은 흔적이 있으면 "키워드 매칭률은 높지만 부자연스럽습니다 — 인터뷰 노트의 실제 내용을 기반으로 다시 작성하세요"
- JD 미제공 시: "차별화 포인트가 약합니다 — 인터뷰 노트의 '의사결정 근거(Q12)'를 해결 섹션에 추가하세요."

## 카테고리 3: 가독성 (Readability)

0~100점. 임계 75점.

### 채점 기준

| 하위 항목 | 배점 | 평가 |
|---|---|---|
| 길이 적정성 | 30 | 도입부 (50~80 단어 / bullet 3~5개), 본문 섹션 (80~150 단어). 각 권장 ±20% 이내면 만점 |
| 문장 흐름 | 30 | 한 문장의 길이, 능동태/수동태 균형, 단락 간 연결 |
| 1차 스캔 가독성 | 25 | 채용 담당자가 30초 안에 핵심을 파악할 수 있는가 — 굵은 표시, bullet 활용, 핵심 문장 위치 |
| 한국어 자연스러움 | 15 | 번역투, 어색한 어휘, 일관된 어투(존댓말 vs 평어체), 구어체 혼용 여부 |

### 통과 예시

```markdown
**해결**

- MSW로 API 스키마를 프론트엔드에서 선제 정의하고 백엔드에 역제안하는 워크플로우를 도입했다.
- Orval 도입으로 OpenAPI 변경 시 클라이언트 SDK가 자동 재생성되도록 했다. `react-query` 통합이 기본 제공되어 보일러플레이트까지 흡수했다.
```

→ 2개 bullet, 각 한 문장으로 압축. 능동태. 한국어 자연스러움. 약 80 단어로 권장 범위 내.

### 미통과 예시

```markdown
**해결**

- 우리는 매우 어려운 상황에서 어떻게든 해결책을 찾아야만 했다. 그래서 우리는 다양한 도구들을 검토하기 시작했고 그중에서 MSW라는 도구를 사용하게 되었는데 이 도구는 모킹을 위한 도구이며 우리는 이것을 통해서 백엔드 API가 완성되기 전에도 프론트엔드 개발을 진행할 수 있게 되었다. 그리고 또한 Orval이라는 도구도 있었는데...
```

→ 한 문장이 너무 길고 수동태. 200 단어 초과. 핵심이 묻혀 1차 스캔 어려움. 가독성 50점 수준.

### 피드백 작성 가이드

- 길이 위반 명시: "해결 섹션이 7문장으로 권장(3~5문장) 초과. 'Orval 도입' 부분을 한 문장으로 압축."
- 문장 구조 지적: "수동태 비율이 높습니다 — '도입되었다' → '도입했다'로 능동 전환."
- 어투 지적: "구어체('것 같아요')가 섞여 있습니다 — 평서체로 통일."

## 종합 점수 계산

```
종합 점수 = (구조 × 0.30) + (적합도 × 0.35) + (가독성 × 0.35)
```

- 구조가 0점이면 종합 점수와 무관하게 자동 FAIL (구조 Binary 임계 미통과).
- 카테고리별 임계 통과 + 종합 80점 이상이어야 PASS.

## 출력 형식

평가 결과는 다음 형식으로 출력한다.

```markdown
## Phase 4 평가 결과

**판정**: PASS / FAIL
**종합 점수**: 84 / 100
**평가자 시점**: IT 채용 담당자 (--reviewer=hr 기본값)

### 카테고리별 점수

| 카테고리 | 점수 | 임계 | 통과 |
|---|---|---|---|
| 구조 | 100 / 100 | Binary | ✅ |
| 적합도 | 78 / 100 | 70 | ✅ |
| 가독성 | 80 / 100 | 75 | ✅ |

### 강점
- 결과에 "개발 기간 50% 단축"이라는 명확한 정량 지표 포함.
- JD의 Turborepo, FSD 키워드가 자연스럽게 녹아 있음.

### 개선 가능 (FAIL이 아니지만 다음 회차에 참고)
- 가독성: 해결 섹션 마지막 문장이 25 단어로 약간 길다. 가능하면 두 문장으로 분리.
```

FAIL인 경우 출력 형식:

```markdown
## Phase 4 평가 결과

**판정**: FAIL
**종합 점수**: 65 / 100
**평가자 시점**: IT 채용 담당자

### 카테고리별 점수

| 카테고리 | 점수 | 임계 | 통과 |
|---|---|---|---|
| 구조 | 0 / 100 | Binary | ❌ |
| 적합도 | 78 / 100 | 70 | ✅ |
| 가독성 | 82 / 100 | 75 | ✅ |

### 미통과 카테고리: 구조

**문제**: 결과 섹션에 정량 지표가 없습니다.

**구체적 피드백**:
- 인터뷰 노트의 "Q6 답변: 빌드 시간 70% 단축"을 결과 섹션에 명시하세요.
- 부수 효과로 답한 "팀 신뢰도 향상" 부분은 정량화 어려운 임팩트 슬롯으로 추가 가능합니다.

### 재작성 지시
구조 미통과를 우선 해결한 후 다시 평가합니다. (재작성 시도 1/3)
```

## 기술 면접관 시점 (`--reviewer=tech` 또는 `--reviewer=both`)

기본 IT 채용 담당자 시점에 더해 기술 면접관 시점을 추가한다. 두 평가자는 평가 우선순위가 다르다.

| | IT 채용 담당자 (기본) | 기술 면접관 |
|---|---|---|
| 핵심 관심사 | 키워드 필터, 가독성, 분량, ATS 통과 | 기술 깊이, 트레이드오프 인식, 의사결정 근거 |
| 적합도 평가 시 | JD 키워드 매칭 비중 ↑ | 의사결정 근거의 합리성 비중 ↑ |
| 가독성 평가 시 | 1차 스캔 가독성 비중 ↑ | 기술 정확성 비중 ↑ |
| 가산점 항목 | 임팩트 정량화 | 실패·재발견 서사, 외부 제약 인식 |

`--reviewer=both` 옵션이면 두 시점 점수를 별도 표로 출력하고, **둘 다 PASS여야 종합 PASS** 처리한다.

## 재작성 루프 한도

- 최대 3회 재작성. 3회 후에도 미통과 시 SKILL.md가 사용자에게 보고:
  - 현재까지의 모든 시도 점수
  - 가장 자주 미통과한 카테고리
  - 사용자 결정 요청: "(a) 강제 진행하여 drafts/ 저장 / (b) 인터뷰 보강 후 처음부터 / (c) 종료"
- 재작성 시 이전 사이클 점수를 명시해 진행 추이 시각화.
